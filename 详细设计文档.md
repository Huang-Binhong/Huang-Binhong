- # 数字 黄宾虹 网站 — 详细设计文档


  > *版本：v1.0
  > *日期：2025-11-11*
  >

  ---

  ## 1. 项目背景 & 目标

  ### 1.1 背景

  本项目旨在打造一个以“数字 黄宾虹”为主题的线上展览／知识平台，参照 上海博物馆 的“数字 董其昌”形式，通过数字化技术呈现黄宾虹的人物生平、艺术成就、作品专题、理论风格与时代背景。

  ### 1.2 目标

  - 为用户提供一个清晰、结构化、互动性强的黄宾虹专题网站。
  - 通过前端交互、后台数据＋API、内容管理三者协作，实现内容的便捷更新、访问体验优化。
  - 突出“人物生平”“人际关系”“作品专题”“艺术理论与风格”“时代背景与环境”五大板块。
  - 界面美观、响应式设计，适用于桌面和移动设备。
  - 后端具备内容管理（CMS或自建）、API服务、数据持久化、安全性等。

  ## 2. 功能需求

  ### 2.1 用户角色

  | 角色     | 描述           | 权限                                 |
  | -------- | -------------- | ------------------------------------ |
  | 访客用户 | 普通网站访问者 | 浏览所有公开内容、筛选作品、搜索查询 |

  > **说明**：本项目为纯展示型网站，所有内容通过数据库预先录入，后端仅提供只读 API（GET 请求），无需用户登录、编辑和内容管理功能。
  >

  ### 2.2 关键用例

  - 用户访问首页，选择板块进入。
  - 用户在"人物生平"板块查看时间轴、浏览生平事件。
  - 用户在"人际关系"板块查看交往关系图谱、点击人物查看详情。
  - 用户在"作品专题"板块浏览作品列表，使用筛选和搜索功能。
  - 用户点击作品查看详情（高清图＋创作背景＋艺术分析）。
  - 用户在"人物行迹"板块查看交互式地图，探索不同地点的相关内容。

  ### 2.3 功能清单（按模块）

  #### 导览／首页

  - 首页 Banner 大图 + 导语。
  - 各板块入口（人物生平、人际关系、作品专题、人物行迹）。
  - 搜索／筛选入口（按关键词、作品名、题材等）。
  - 使用说明／交互提示（例如："可点击卡片进入"、"可放大查看作品细节"）。

  #### 人物生平

  - 时间轴组件：按年份排列黄宾虹的重要人生节点。
  - 地点互动图：展示其出生、学习、创作、旅居地点。
  - 分阶段内容：早年／中年／晚年，每段包含家庭背景、教育经历、职业转折、创作高峰、晚年影响。
  - 图像／影像资料：旧照片、手稿、信札、故居图片。

  #### 人际关系／艺术圈交往

  - 关系图谱（节点-连线图）：人物／机构及其关系。
  - 师承关系子页（其师、影响他的人）。
  - 弟子／学生子页（其学生、后续影响者）。
  - 合作机构／画院／展览机构子页。
  - 交往案例专题（如一次重要画展、一次鉴赏合作、旅行写生活动）。

  #### 作品专题

  - 作品浏览页：按时间／题材／材质／风格进行分类与筛选。
  - 代表作解析页：每件作品的创作时间、尺寸、材质、题款、印章、风格特征、历史意义。
  - 技法分析页：笔法、墨色、色彩、构图、题款／印章特征。
  - 放大镜功能：用户可放大查看作品细节。

  #### 艺术理论与风格

  - 艺术观点／讲稿／随笔摘录：如黄宾虹对传统山水画的理解、书法与绘画关系的论述。
  - 风格演变：早期／中期／晚期画风变化说明。
  - 创新与传统对比：其如何继承中国山水画传统、提出何种创新。
  - 技法专题：如“泼彩法”“云树皴擦法”“墨色变化”“构图大格局”。
  - 多媒体演示（预留）：笔法动画、构图过程动画。

  #### 时代背景与环境

  - 历史时代概况：清末、民国初年，中国传统美术与现代美术交替期的文化环境。
  - 地域文化背景：如浙江金华、歙县、扬州地域及山水画传统。
  - 社会／政治／经济因素：如维新运动、革命时期、社会变动对艺术的影响。
  - 自然地理与旅行写生背景：黄宾虹旅行路线、自然山水对其创作的影响。
  - 技术与媒材环境：当时纸本、水墨、设色、印章、写生工具的制作与使用。

  ### 2.4 非功能需求

  - 响应式 (RWD) 设计：兼容桌面、平板、手机。
  - 性能要求：页面加载时间短，图片优化加载。
  - 可维护性：模块化架构，便于后续内容扩展。
  - 可扩展性：预留 VR／3D／元宇宙模块接口。
  - 可访问性：遵循基本 WCAG 标准（如图片 alt 属性、键盘导航）

  ## 3. 系统架构设计

  >

  ### 3.2 模块划分

  - 前端模块：

    - Home 模块
    - PersonBio 模块（人物生平）
    - RelationGraph 模块（人际关系）

    - API 模块（提供只读数据接口）

    - Person（人物）

    - Event（生平事件）
    - Location（地点）
  - ### 3.3 接口设计

    #### 接口设计原则

    按照 RESTful 设计最佳实践：资源为主、路径用名词、仅使用 GET 方法、支持分页／过滤、状态码语义明晰。


    - 路径使用复数名词，如 `/persons`、`/works`。
    - 仅使用 `GET` 方法读取资源（无需 POST、PUT、DELETE）。
    - 接口应为无状态（每次请求都包含所需信息）以保证可扩展性。
    - 支持分页、过滤、排序，以便查询大量数据而不导致性能问题。
    - 错误响应应使用标准 HTTP 状态码，并提供结构化错误信息。
    - API 版本管理（如 `/api/v1/…`）以保证未来可变更。

    ---

    #### 通用响应格式（示例）

    ```
    {
      "code": 0,
      "message": "success",
      "data": { … }
    }
    ```
    - `code`：整型，0 表示成功，非 0 表示错误
    - `message`：文本，成功或错误信息
    - `data`：对象、数组或 null，具体接口返回数据

    #### 错误码规范（示例）

    | HTTP 状态码 | code | 描述         |
    | ----------- | ---- | ------------ |
    | 400         | 4001 | 请求参数错误 |
    | 404         | 4041 | 资源不存在   |
    | 500         | 5001 | 服务器错误   |

    ---

    #### 接口清单（主要资源）

    以下为各资源接口示例，所有接口均为 **GET 只读接口**：

    **1. 人物 (Person)**

    - `GET /api/v1/persons`

      - 描述：获取人物列表
      - 查询参数（可选）：`page`（int，默认 1）、`pageSize`（int，默认 20）、`search`（字符串，按姓名或别号模糊）
      - 返回：分页对象，`items` 为 Person 对象数组
    - `GET /api/v1/persons/{personId}`

      - 描述：获取某个人物详情
      - 路径参数：`personId`（字符串）
      - 返回：Person 对象
        **2. 生平事件 (Event)**
    - `GET /api/v1/persons/{personId}/events`

      - 描述：获取某人物的生平事件列表
      - 路径参数：`personId`
      - 返回：事件数组，按 year 升序
    - `GET /api/v1/events/{eventId}`

      - 描述：获取事件详情
      - 返回：Event 对象

    **3. 地点 (Location)**

    - `GET /api/v1/locations`
      - 获取所有地点列表
    - `GET /api/v1/locations/{locationId}`
      - 获取某地点详情

    **4. 人际关系 (Relation)**

    - `GET /api/v1/relations`
      - 获取所有关系
      - 支持查询参数：`type`（关系类型）、`fromPersonId`、`toPersonId`
    - `GET /api/v1/persons/{personId}/relations`
      - 获取指定人物的所有关系
    - `GET /api/v1/relations/{relationId}`
      - 获取关系详情

    **5. 作品 (Work)**

    - `GET /api/v1/works`
      - 获取作品列表
      - 查询参数：`page`、`pageSize`、`category`、`stylePeriod`、`material`、`search`
      - 返回：分页对象，`items` 为 Work 对象
    - `GET /api/v1/works/{workId}`
      - 获取作品详情

    ---

    #### 分页结构示例

    ```
    {
      "code": 0,
      "message": "success",
      "data": {
        "total": 345,
        "page": 2,
        "pageSize": 20,
        "items": [ … ]
      }
    }
    ```
    ---

    #### 常用查询参数说明

    - `page`: 整数，当前页码，默认 1
    - `pageSize`: 整数，每页条数，默认 20
    - `search`: 字符串，模糊搜索关键词（如姓名、作品标题）
    - `category`, `stylePeriod`, `material`: 字符串，作为筛选条件
    - `sortBy`: 字符串，指定排序字段，如 `creationDate_desc` 或 `title_asc`

  ## 4. 前端设计

  ### 4.1 信息架构 & 路由设计

  #### **1. 整体信息架构与路由设计**

  网站的整体信息架构已经更新，以提供更清晰、更专注的用户浏览路径。具体的页面路由设计如下：

  - `/` — 首页
  - `/bio` — 人物生平
  - `/relations` — 人际关系总览
  - `/relations/person/:id` — 某人物交往详情
  - `/works` — 作品专题浏览页
  - `/works/:id` — 作品详情页
  - `/experience` — 人物行迹
  - #### **2. 各页面模块详细设计**

    ##### **2.1 首页 (`/`)**


    - **功能描述**: 作为网站的门户，快速引导用户至各个核心板块，展示关键内容的精粹。
    - **包含内容**:
      - 核心人物的简要介绍入口。
      - 精选作品的瀑布流或轮播展示。
      - 人际关系网络的可视化预览图。
      - 人物行迹地图的动态缩影。
      - 清晰的顶部导航栏，链接至所有一级页面。

    ##### **2.2 人物生平 (`/bio`)**

    - **功能描述**: 通过时间线的形式，全面、深入地介绍人物的生平、重要经历和时代印记。
    - **包含内容**:
      - **生平时间轴**: 按年代划分，展示人物在不同阶段的关键事件、个人成长和思想变迁。
      - **时代背景**: 在时间轴的相应节点，无缝融入当时的社会、历史和文化背景介绍，帮助用户更深刻地理解人物的行为与选择。例如，在介绍青年时期的经历时，会同步介绍当时的社会思潮和历史事件。
      - 图文资料: 包含各时期的照片、书信手稿等珍贵资料。

    ##### **2.3 人际关系**

    - **2.3.1 人际关系总览 (`/relations`)**
      - **功能描述**: 提供一个宏观、可视化的关系网络图谱，让用户直观地了解人物的社交圈。
      - **包含内容**: 以核心人物为中心的关系图，展示其与家人、朋友、同事、对手等的关系。用户可以点击不同的人物节点，查看简介并跳转至详情页。
    - **2.3.2 某人物交往详情 (`/relations/person/:id`)**
      - **功能描述**: 聚焦于核心人物与另一位特定人物的双边关系，深入挖掘二者之间的交往故事。
      - **包含内容**: 两位人物的交往时间线、关键的共同事件、信件往来内容、相互评价、合作的作品等。

    ##### **2.4 作品专题**

    - **2.4.1 作品专题浏览页 (`/works`)**
      - **功能描述**: 系统性地展示人物的全部或主要作品，支持用户进行分类浏览和筛选。
      - **包含内容**:
        - 作品画廊/列表展示。
        - 筛选功能: 可按作品类型、创作年份、主题等维度进行筛选。
        - 搜索功能: 支持用户按关键词搜索特定作品。
    - **2.4.2 作品详情页 (`/works/:id`)**
      - **功能描述**: 对单一作品进行全方位的深度剖析和解读。
      - **包含内容**:
        - 作品高清数字内容（图片、文本、音视频等）。
        - 作品基本信息: 创作年代、材质、尺寸、现藏地等。
        - 创作背景故事: 介绍作品诞生的缘由和过程。
        - 在此部分详细阐述作品所体现的艺术思想、创作理念、使用的技巧和风格。同时，会包含对作品在艺术史上的地位、影响以及后世评价的分析。

    ##### **2.5 人物行迹 (`/experience`)**

    - **功能描述**: 结合地理信息与时间线，动态、直观地展示人物一生的足迹。
    - **包含内容**:
      - 一个可交互的地图，标注出人物曾居住、游历或发生重要事件的地点。
      - 点击地图上的标记点，可查看与该地点相关的生平事件、创作的作品或交往的人物。
      - 可与 `/bio`页面的时间轴进行联动，实现时空同步展示。

  ### 4.2 UI & UX 概要

  - 顶部导航（Header）：项目 logo、菜单按钮（四大板块）、搜索框。
  - Banner 区域：大图 + 标题 + 按钮（例如"进入专题"）。
  - 模块卡片：用于各板块入口，卡片式 UI，鼠标／触摸交互。
  - 面包屑导航（Breadcrumb）提示用户当前位置。
  - 响应式布局：大屏为多列显示，移动设备为单列滚动。
  - 图像优化：作品详情页使用懒加载、大图可放大查看。
  - 关系图谱：使用 D3.js 或 vis.js 绘制节点-连线图，可拖动、缩放。
  - 时间轴组件：横向滚动或纵向排列，明确标出年份与事件。
  - 地图交互：使用 Leaflet 或 Google Maps API 标注地点。

  ### 4.3 样式指南（初步）

  - 主色调：墨黑 (#2E2E2E)、墨绿色 (#3B4F3A)、米白 (#F7F5F0)
  - 辅助色：金黄 (#D4A451)（用于强调、提示）
  - 字体：标题使用“思源黑体”／“Source Han Sans”，正文使用“苹方”／“PingFang SC”
  - 图标：简约线性图标 (Icon font)
  - 按钮样式：圆角 4px，悬停状态颜色稍微变深。
  - 间距：统一 8px 为基础单位，布局采用其倍数。

  ### 4.4 前端状态管理 & 数据流

  - 使用 Redux 或 Vuex 管理全局状态（如筛选条件、当前浏览的作品等）。
  - 各模块组件通过 API 拉取数据，存储至状态管理库，避免重复请求。
  - 异步操作使用 async/await 或 Saga/Thunk。
  - 错误／加载状态统一展示（Loading Spinner、错误提示）。

  ### 4.5 前端测试策略

  - 单元测试：组件级别使用 Jest + React Testing Library 或 Vue Test Utils。
  - 集成测试：关键流程（如作品详情加载、关系图谱渲染）使用 Cypress 或 Playwright。
  - UI 快照测试：关键页面使用 Snapshot 测试确保 UI 无意更改。

  ## 5. 后端设计

  ### 5.1 架构概览

  - 采用 REST 架构风格，前后端分离。
  - 所有接口均为只读（GET 请求），无需身份认证。
  - 日志与监控：记录 API 访问日志、异常记录。
  - 静态资源：作品图片、地图标记图片等存储在对象存储（如 AWS S3）或 CDN。

  ### 5.2 数据库设计（示例）

  - Person 表：id、name、birthDate、deathDate、briefIntro、portraitUrl 等。
  - Event 表：id、personId、year、title、description、locationId 等。
  - Relation 表：relationId、fromPersonId、toPersonId、relationType、description。
  - Work 表：id、title、creationDate、material、size、imageUrls、description、technique、category、stylePeriod。
  - Location 表：locationId、name、coordinates、imageUrl、description。

  ### 5.3 API 规范（简要）

  - 请求／响应统一格式（例如：`{ code: 0, message: "success", data: … }`）。
  - 错误码统一定义（如：4001 请求参数错误；4041 资源不存在）。
  - 分页机制：列表接口支持 page 和 pageSize 参数。
  - 所有接口均为 GET 请求，无需认证。

  ### 5.4 数据管理

  - 内容数据通过数据库迁移脚本或批量导入工具预先录入。
  - 图片资源存储在 CDN 或对象存储服务，数据库仅保存 URL。
  - 定期备份数据库与媒体资源。

  ### 5.5 后端测试策略

  - 单元测试：使用 Mocha/Jest 测试业务逻辑模块。
  - 接口测试：使用 Postman 或 Supertest 验证 API 正确性。
  - 性能测试：关键接口（如作品列表、作品详情）响应时间监控。

  ## 6. 数据／内容管理

  ### 6.1 内容结构定义

  - 人物：姓名、别号、出生地、简介、照片、生平事件。
  - 关系：人物-人物／人物-机构关系，关系类型标签，描述。
  - 作品：题目、创作时间、材质、尺寸、图像、解读文字。
  - 地点：名称、经纬度、简介、相关人物或作品。

  ### 6.2 数据导入方式

  - 使用数据库迁移脚本（如 SQL 脚本、MongoDB 导入工具）批量导入数据。
  - 图片资源上传至 CDN 或对象存储，获取 URL 后写入数据库。
  - 可编写数据导入工具（Node.js 脚本、Python 脚本）读取 Excel/CSV 文件批量导入。

  ### 6.3 内容更新流程

  1. 内容更新通过数据库脚本或后台工具直接操作数据库。
  2. 静态资源更新后，需清除 CDN 缓存。
  3. 定期备份数据库与媒体资源。
